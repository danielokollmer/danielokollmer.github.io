Phase 3 — Microsoft Store (0€)

250 Mio. monatl. aktive Nutzer · Gratis seit Sep 2025 · PWABuilder
0€ Invest
01

Developer-Account erstellen (kostenlos)

→ storedeveloper.microsoft.com · Persönliches Microsoft-Konto verwenden. Seit September 2025 keine Registrierungsgebühr mehr für Einzelpersonen.

02

App-Namen reservieren im Partner Center

Partner Center → "Neues Produkt" → "MSIX oder PWA App" → Name reservieren: z. B. "Interaktives Periodensystem". Publisher-ID und Package-ID notieren.

03

PWA packen mit PWABuilder

→ pwabuilder.com · GitHub-Pages-URL eingeben → Evaluierung abwarten → "Package for Stores" → Windows → ".msixbundle" herunterladen.

URL eingeben: https://DEIN-USERNAME.github.io/periodensystem/ → "Start" → Report Card ≥ 70% → "Package For Stores" → Windows → "Generate Package" → Publisher Name, Publisher ID aus Partner Center eintragen → "Download Package" (.zip mit .msixbundle + .classic.appxbundle)
04

App im Partner Center einreichen

Partner Center → App auswählen → "Submission starten" → Preis, Altersfreigabe, Screenshots, Store-Beschreibung ausfüllen → .msixbundle hochladen → Einreichen. Review: 24–48 Stunden.

Commerce-Option: Für Nicht-Spiele-Apps kann Microsoft's eigenes Bezahlsystem oder ein externes (z. B. Payhip-Link) verwendet werden. Externes System → 100% Erlös (Stripe/PayPal-Gebühr fällt trotzdem an).

Phase 4 — Google Play Store (25€)

TWA (Trusted Web Activity) · Bubblewrap / PWABuilder · 15% Commission
25€ einmalig
01

Google Play Developer Account erstellen

→ play.google.com/console · Einmalige Registrierungsgebühr: 25 USD. Google-Konto erforderlich. Danach unbegrenzt viele Apps veröffentlichbar.

02

Android-Package mit PWABuilder generieren

→ pwabuilder.com → gleiche URL wie für Microsoft → "Package for Stores" → Android. PWABuilder erzeugt eine .aab-Datei (Android App Bundle) via TWA-Technologie — der Browser lädt deine gehostete PWA, sieht aber wie eine native App aus.

Package Name: de.deinname.periodensystem # z.B. de.dakocreatives.pse App Version: 1 Display: standalone Signing: Neuen Keystore erstellen lassen → SHA-256-Fingerprint notieren!
Keystore sichern! Die generierte .jks-Keystore-Datei und das Passwort SICHER aufbewahren (z. B. KeePass, verschlüsseltes Backup). Ohne Keystore können Updates nicht veröffentlicht werden.
03

assetlinks.json mit SHA-256 befüllen

Den SHA-256-Fingerprint aus dem Keystore in .well-known/assetlinks.json eintragen. Dann zu GitHub pushen. Ohne diese Datei zeigt die App im Browser eine Adressleiste — mit ihr verschwindet sie und es sieht nativ aus.

{ "relation": ["delegate_permission/common.handle_all_urls"], "target": { "namespace": "android_app", "package_name": "de.dakocreatives.pse", "sha256_cert_fingerprints": [ "XX:XX:XX:...:XX" ← aus Keystore ] } }
04

App in Google Play Console einreichen

Play Console → "App erstellen" → .aab hochladen → Store-Eintrag (Beschreibung DE+EN, Screenshots, Icon, Datenschutzerklärung-URL) → "Interne Tests" zuerst, dann "Produktion". Review: 3–7 Werktage beim ersten Mal.

Datenschutzerklärung: Pflicht bei Google Play. Da die App keine personenbezogenen Daten verarbeitet und alles lokal läuft, reicht eine sehr kurze Erklärung. Kann als GitHub Pages-Seite gehostet werden.

01

Aufwand vs. Ertrag abwägen

Erfordert: Apple Developer Account (99 USD/Jahr), Mac (für Xcode), Signierungszertifikate, App Store Review (7–14 Tage). PWABuilder kann ein iOS-Package erzeugen, aber Apple-Review ist strenger als Google/Microsoft.

Für den DACH-Bildungsmarkt: iPad ist verbreitet in Schulen → könnte sich mittelfristig lohnen. Empfehlung: erst Phase 3+4 stabilisieren, dann iOS evaluieren.
